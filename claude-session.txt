  1. navigator at module level (line 50) — the main issue

  const PASSWORD_CHAR = navigator.userAgent.match(/firefox|fxios/i)
    ? "\u25CF"
    : "\u2022";

  This runs at module load time. In Next.js (even with "use client"), the module
   is evaluated during SSR where navigator doesn't exist → ReferenceError: 
  navigator is not defined. Fix it with a lazy getter or typeof guard:

  const getPasswordChar = () =>
    typeof navigator !== "undefined" &&
  navigator.userAgent.match(/firefox|fxios/i)
      ? "\u25CF"
      : "\u2022";

  Then replace PASSWORD_CHAR usage with getPasswordChar() (or call it once
  inside the component).

  2. params.spring type mismatch (line 115–120)

  The object stored in useDialKit under spring has type: "spring" (a DialKit
  discriminator), but useSpring from framer-motion expects SpringOptions, which
  has no type field. TypeScript will flag params.spring as incompatible. Fix:

  const { type: _springType, ...springOptions } = params.spring;
  const springCaretX = useSpring(
    caretX,
    prefersReducedMotion
      ? { stiffness: 10000, damping: 100, mass: 0.1 }
      : springOptions,
  );

  3. dialkit package — if it's not in package.json or not installed, the import
  on line 3 will be red. Verify with:

  npm ls dialkit

  The navigator at module scope (issue 1) is the most likely cause of the
  visible red lines, since it's a server/client environment mismatch that
  TypeScript or the Next.js linter would flag.
